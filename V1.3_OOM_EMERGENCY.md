## Prime suspects (in order)

1. **Gemma → Bark VRAM handoff in `nodes/gemma4_orchestrator.py` and `nodes/batch_bark_generator.py`.** Gemma 4 is the heaviest single model in the pipeline. If `del model`, `gc.collect()`, and `torch.cuda.empty_cache()` are not called in the right order (or are skipped on certain code paths), Bark will load on top of a dirty allocator and blow the 14.5 GB ceiling.
2. **Arc Enhancer Phase A/B/C re-entry into Gemma.** Phase B bookend rewrite may be loading Gemma a second time without fully unloading it from Phase A. Check `_arc_check_and_rewrite_bookends` for nested load paths.
3. **SageAttention + SDPA fallback.** On some prompts the attention backend can silently fall back to a higher-memory path. Check the startup log for backend reports.
4. **OpenClose evaluator.** Gated off in v1.3 (`ENABLE_3_OUTLINE_EVALUATOR = False`) — confirm it really is off in the failing run.
5. **Allocator fragmentation.** Even with correct `empty_cache()`, the torch allocator can fragment across a long multi-node run. `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True` is the standard mitigation on recent torch.

## Field report — observed failure signature (read this)

A prior run of the v1.3 full workflow exhibited three overlapping failure classes. Treat this as eyewitness data, not speculation.

### Three failure classes observed
1. **Script structural drift.** The generated screenplay itself was unstable: names, scene labels, and dialogue formatting broke apart mid-stream. Suggests the prompt or postprocessor is letting the model emit unbounded, partially structured text.
2. **Critique/revision loop non-convergence.** The revision pass timed out and fell back to the original draft. The fallback path may be injecting timeout/error text into script output, corrupting downstream parsing.
3. **Hard CUDA OOM on model load.** Crash occurred during model *loading*, not sampling. VRAM was already sitting at **31.70 GiB allocated** (well over the 14.5 GB target and over the 16 GB hardware ceiling — implies prior models were never released) when the next load attempted. This is the blocker symptom.

### QA checklist for the OOM hunt

**1. Structural integrity of script output**
- Every scene starts and ends cleanly.
- Speaker tags remain valid and consistent.
- Parenthetical beats and SFX lines do not interleave into dialogue text.
- XML-like blocks such as `<vocal_blueprints>` and `<locked_decisions>` stay intact and do not leak into the script body.

**2. Dialogue quality**
- No long expository lines that read like summary rather than spoken dialogue.
- Flag repeated abstract phrases ("traction," "throughput," "integrity," "viability") if ungrounded.
- Character voices stay distinct across scenes.
- Character names spelled identically everywhere.

**3. Continuity and logic**
- Scene-to-scene transitions have a clear inciting action.
- Emotional stakes escalate naturally.
- Ending follows from the failure in the climax scene instead of being undone by an unrelated success announcement.
- Technical/medical claims used consistently, no contradictions with earlier setup.

**4. Prompt and formatting robustness**
- Model is not allowed to regenerate the same section multiple times in different styles.
- Streaming output is not appended without proper truncation or delimiter handling.
- Fallback behavior on revision timeout is contained — does not contaminate script text.
- Content filters do not replace words so late they corrupt surrounding formatting.

**5. System stability (PRIMARY OOM FOCUS)**
- Model is NOT being reloaded unnecessarily between passes.
- GPU memory is released before loading the next pass (`del model` + `gc.collect()` + `torch.cuda.empty_cache()` in that exact order).
- Multiple outputs are not being held in memory at once.
- Context window is not growing from repeated critique/rewrite loops.

### Concrete errors to test for
- Scene text corruption (words from different sections interleaved).
- Duplicate generation of the same scene in conflicting forms.
- Character-name drift (inconsistent spelling, accidental renaming).
- Revision timeout causing fallback to the original draft.
- **OOM during model load, especially when a large model is reloaded after earlier passes. This is the blocker.**
- Formatting breakage caused by injected timeout/error text inside script output.
- Final resolution mismatch (epilogue contradicts the dramatic outcome).

### Likely fixes (practical, not literary)
- Shorten prompts and reduce retained intermediate text.
- Stop re-inserting the full draft into later passes unless absolutely necessary.
- Add a hard output delimiter so tool logs cannot contaminate script text.
- Lower model load pressure: unload every prior model before the next pass.
- Reduce max-token budgets for revision and echo stages.
- Split the workflow so scene writing, critique, and final polish happen as separate bounded jobs.
- Add a validation step that rejects malformed output before it reaches the next stage.

### Batching strategy for the rewrite
1. Single pass to extract the stable script only.
2. Separate QA pass that flags structural, dialogue, and continuity issues.
3. Repair pass only on flagged segments, not the full draft.
4. Load the final model once, after all earlier models are fully freed.
5. Keep the approval epilogue as a separate postscript, not inside the script body.

### Inspect first
Start with **memory and output containment**. The OOM showed the system sitting at 31.70 GiB allocated when it tried to load another model, which is mathematically impossible on a 16 GB card unless something is double-counting or holding stale allocations. Freeing VRAM and preventing redundant reloads is priority one. After that, fix the text pipeline so scene content cannot be corrupted by timeout messages or fallback output.
